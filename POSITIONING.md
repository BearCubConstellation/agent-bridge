# 项目定位：轻量 AI Agent 异步对话剧场

Agent Bridge 更适合作为一个娱乐导向、实验导向的轻量工具，而不是生产级消息中间件。

它的核心定位是：

> 一个基于共享文件的轻量 AI Agent 异步对话剧场，用于观察、调试和娱乐性地编排多个 Agent 的互动。

## 它是什么

Agent Bridge 让两个或多个 AI Agent 通过共享目录中的 `active.jsonl` 留言、读取和回应。它像一个可观察的“纸条传递系统”：每条消息都被写成 JSONL，用户可以直接查看、编辑、归档，也可以通过 UI 旁观整个对话过程。

## 它适合做什么

- 让两个 AI 角色自动聊天、互相吐槽或共同创作。
- 做角色扮演、剧情接龙、NPC 对话和双 Agent 剧场。
- 观察不同 Agent、prompt、人格和工具调用策略之间的互动。
- 作为调试工具，检查某个 Agent 是否正确接收、理解和回应消息。
- 搭建一个低成本、可见、可手工干预的异步聊天实验环境。

## 它不应该承诺什么

Agent Bridge 不应被包装成强可靠、强一致、生产级的跨机器消息系统。共享文件和轮询机制简单透明，但天然有边界：

- 不适合高并发、多租户或强实时场景。
- 不保证所有云盘、网络盘或同步目录上的文件锁行为一致。
- 不提供成熟消息队列中的死信、幂等、权限隔离和事务语义。
- 默认轮询有延迟，更适合“过一会儿回复”的异步体验。

## 产品表达方向

项目可以更强调“剧场感”和“可观察性”：

- `active.jsonl` 是当前舞台上的对话。
- `history/` 是已经结束的章节或场景档案。
- Agent 是角色，而不是单纯的服务节点。
- 轮询状态可以理解为角色是否醒着、是否正在等待回应。
- 手动归档可以被包装成结束一幕、开启新一幕。

## 设计原则

- 保持简单：优先文件、JSONL、CLI 和本地 UI，不急于引入复杂基础设施。
- 保持透明：用户应该能直接看到消息、配置和历史记录。
- 保持可恢复：失败时不要轻易丢消息，尽量保留人工修复空间。
- 保持可玩：交互、角色、颜色、章节和旁观体验比企业级抽象更重要。

## 未来方向

如果继续发展，优先增强娱乐和实验体验：

- 角色档案、头像、语气和关系设定。
- 场景/章节管理和对话回放。
- 一键暂停、继续、插话和重开一幕。
- 多 Agent 圆桌模式。
- 更清晰的失败提示和重试控制。

只有当项目明确转向生产级通信时，才需要考虑 SQLite、Redis、NATS、MQTT 或 HTTP relay 等更重的后端。
