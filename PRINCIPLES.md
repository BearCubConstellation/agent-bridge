# Agent Bridge — 开发原则

> 任何 AI 或人类在修改本项目代码之前，必须先读完本文件。

---

## 一、这个项目是什么

Agent Bridge 是一个 **Agent 间异步通信中间件**。

它让两个独立运行的 AI Agent 通过共享文件系统进行异步对话。不依赖消息队列、数据库、WebSocket 或任何外部服务——唯一的传输介质是文件。

**一句话概括：两个 Agent 各自往同一个文件里写消息，定时检查有没有对方的新消息，有就通知自己。**

### 它解决什么问题

两个 Agent 需要持续对话，但它们的运行节奏不同、可能不在同一台机器上、可能不会同时在线。直接 API 调用会紧耦合，消息队列引入外部依赖，实时连接要求双方同时在线。

Agent Bridge 用最简单的方式解决：写文件。文件不会丢，不需要同时在线，不需要额外基础设施。

### 核心工作方式

1. Agent A 想说话 → 往共享文件追加一条消息
2. Agent B 的定时任务检查文件 → 发现新消息
3. 通过 HTTP 通知 Agent B 去处理
4. Agent B 处理完 → 往同一个文件追加回复
5. Agent A 的定时任务下一轮发现回复 → 通知 Agent A

没有新消息时，检查脚本直接退出，不消耗任何 LLM token。

---

## 二、最终目标

这个项目要达成以下目标，所有修改都应该服务于这些目标，任何偏离都需要重新审视：

1. **消息不丢失** — 已经写下的消息，无论发生什么（进程崩溃、网络中断、并发操作），都不能消失
2. **消息不重复投递** — 同一条消息不应该反复通知对方，浪费 LLM token
3. **空闲零成本** — 没有新消息时，轮询不消耗 token、不发起网络请求
4. **双方完全独立** — 任一方可以随时离线，回来后继续从中断处接上
5. **开箱即用** — 不需要安装数据库、消息队列或任何外部服务，Python 标准库即可运行
6. **人可观测** — 所有状态都是普通文件，用系统自带工具就能查看和排查问题
7. **框架无关** — 不绑定任何特定 Agent 框架，任何能收 HTTP 请求和写文件的 Agent 都能接入
8. **容错** — 通知对方失败时不放弃，自动重试；游标只在确认对方收到后才推进，确保消息不会被跳过

---

## 三、绝对禁止

1. **禁止引入外部运行时依赖** — 不用数据库、消息队列、缓存服务。文件系统就是唯一的存储层
2. **禁止使用未经验证的技术方案** — 每一种技术选择必须在 Python 标准库的稳定 API 之内，或者在本项目中有已经跑通的先例。不要因为"看起来更优雅"就引入新的依赖或模式
3. **禁止为了性能牺牲正确性** — 这个项目的核心场景是每 3 分钟轮询一次的低频通信。不需要高性能，需要高可靠。任何优化都不能以增加消息丢失或重复的风险为代价
4. **禁止在可复用模块中做不可逆决策** — 被其他程序导入的模块不能替调用方终止整个进程或做其他不可恢复的操作，必须把决策权交给调用方
5. **禁止把配置分散到多个地方** — 只有一个配置文件，所有模块都从它读取
6. **禁止绑定外网地址** — HTTP 服务只监听本地地址，不暴露到网络
7. **禁止静默吞掉错误** — 出问题时必须有可见的输出，不能假装没发生
8. **禁止假设只有自己在写文件** — 两个 Agent 可能同时写入同一个文件，任何涉及文件写入的操作都必须考虑并发安全
9. **禁止一次改多个关注点** — 一次只改一个问题，不要在同一个修改中同时重构、加功能、改格式。小修改容易验证，大修改容易藏隐患
10. **禁止格式变更不留迁移路径** — 任何消息格式或配置格式的变更都必须让已有数据能继续使用，不能假设用户从零开始

---

## 四、修改前问自己

每次改动前，确认以下问题：

- 这个改动是否让消息更不容易丢失？还是增加了丢失的风险？
- 这个改动是否让系统更简单？还是更复杂？
- 这个改动是否引入了新的外部依赖？
- 如果这个改动出 bug，最坏情况是什么？用户能发现吗？
- 这个改动是否在用一种"聪明"的方式解决一个本不存在的问题？

**如果不确定，就不要改。保持现状优于引入风险。**

---

## 五、什么是好的修改

- 修复了消息丢失或重复的实际 bug
- 让配置、消息格式、归档行为更一致
- 补充测试覆盖之前遗漏的边界情况
- 让文档与实际行为保持同步
- 简化了不必要的复杂度

## 六、什么是危险的修改

- "重构"已经正常工作的并发逻辑
- 为了代码优雅而改变消息格式或文件结构
- 引入新的锁机制或同步策略
- 用异步框架替换同步代码
- 把文件操作换成数据库或缓存
- 合并或拆分模块而不改变实际行为
- 修改归档逻辑而不考虑并发场景 — 归档是消息安全的边界操作，归档时不能丢消息，归档后游标必须正确重置

## 七、改完之后

修改完成后，确认：
- 消息依然能正常收发
- 没有引入新的外部依赖
- 所有现有测试仍然通过

纯文档修改、测试补充、注释修正不需要过以上清单，直接做。
