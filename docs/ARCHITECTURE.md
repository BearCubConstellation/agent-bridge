# Agent Bridge — 架构概览

## 设计原则

1. **文件即协议** — 不依赖任何外部消息队列或数据库
2. **轮询无成本** — 无新消息时零 token、零网络、零 LLM
3. **配置驱动** — 同一套脚本适配任意 agent 框架
4. **UI 为附件** — 核心引擎无头可运行，UI 可选

## 核心抽象

```
┌───────────────────────────────────────────────┐
│                  agent-bridge                   │
│                                                 │
│  ┌──────────┐    ┌────────────┐    ┌──────────┐│
│  │ send.py  │    │  poll.py   │    │ server.py ││
│  │ (写入)    │    │ (轮询唤醒)  │    │ (本地 UI) ││
│  └────┬─────┘    └─────┬──────┘    └─────┬────┘│
│       │                │                 │      │
│       ▼                ▼                 ▼      │
│  ┌──────────────────────────────┐               │
│  │       active.jsonl           │               │
│  │  + 游标文件 + history/       │               │
│  └──────────────────────────────┘               │
└───────────────────────────────────────────────┘
```

## 数据流

1. **发送**：Agent A 调用 `send.py`，向 `active.jsonl` 追加 JSON 行
2. **轮询**：Agent B 的 `poll.py` 每 3 分钟检查文件，发现新消息
3. **唤醒**：`poll.py` POST 到 Agent B 的 webhook/API
4. **回复**：Agent B 处理消息，调用 `send.py`（或等效操作）追加回复
5. **循环**：Agent A 的 `poll.py` 在下一轮发现回复，唤醒 Agent A

## 目录职责

| 目录       | 说明                                         |
|-----------|----------------------------------------------|
| `core/`   | 平台无关的核心逻辑，纯 Python 标准库          |
| `adapters/` | 各 agent 框架的配置模板                    |
| `ui/`     | 本地 Web UI（零依赖，stdlib only）           |
| `setup/`  | 各平台的部署脚本                              |
| `protocol/` | 通信协议规范                              |
| `docs/`   | 详细文档                                      |

## 为什么不是...

### ...API 直接调用？
紧耦合。Agent A 崩溃或离线时消息丢失。文件持久化保证不丢。

### ...消息队列 (RabbitMQ/Redis/NSQ)？
引入外部依赖，增加部署复杂度。文件系统是最简单可靠的持久化层。

### ...WebSocket / SSE？
要求双方同时在线。文件方案天然支持异步——一方写入后离线，另一方下次轮到时处理。
