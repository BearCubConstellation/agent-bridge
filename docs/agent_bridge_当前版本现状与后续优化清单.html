3. 当前已完成能力

3.1 Adapter V2 基础抽象 已完成

已将原来的 core/adapters.py 拆分为 core/adapters/ 目录，支持多种 adapter 类型：

manual native_http openclaw_sessions cli file_mailbox mcp_tool

核心目标：不再依赖旧 wakeup 三元组，能区分不同 Agent 的唤醒方式和回复模式，支持 DeliveryRequest / DeliveryTicket、response.mode、capability 声明，兼容 legacy wakeup。

3.2 Room Runtime V2 状态机 已完成

引入 Runtime V2 状态机，核心状态由 state.json 中的 current_turn 表达。

{
  "status": "running",
  "current_turn": {
    "turn_id": "turn_xxx",
    "agent_id": "openclaw",
    "state": "waiting_response",
    "correlation_id": "corr_xxx",
    "started_at": "2026-xx-xx xx:xx:xx",
    "timeout_at": "2026-xx-xx xx:xx:xx"
  }
}

支持：选择发言 Agent、读取 pending messages、调用 Adapter V2 唤醒、区分 sync/async response、写入 current_turn、等待回写、推进 turn、timeout 检测、legacy 兼容。

3.3 EventBus 事件系统 已完成

新增 events.jsonl 作为系统事件日志，替代了对 runtime.log 和 active.jsonl 的隐式依赖。

message.created agent.wakeup.requested agent.wakeup.succeeded agent.wakeup.failed agent.response.received turn.completed turn.timeout room.started room.paused room.error

3.4 Scheduler 主驱动 已完成

补齐 core/scheduler.py，服务启动时启动 Scheduler，每次 _schedule_room 刷新 config，PollManager 在 V2 模式下降级为 fallback scan，解决了旧 tick_room 和新 Runtime V2 双驱动的风险。

3.5 Callback 回写链路 已完成

新增 Agent 回写入口：POST /api/rooms/{room_id}/agents/{agent_id}/callback

{
  "turn_id": "turn_xxx",
  "correlation_id": "corr_xxx",
  "message": "Agent 的最终回复",
  "meta": {}
}

完整链路：校验 room/agent/turn → append_room_message → 写入 reply_to/correlation_id → emit agent.response.received → schedule 下一轮。解决了 Agent 被唤醒后无统一回写路径导致 waiting_response 死等的核心问题。

3.6 MCP Server 原型 已完成

新增 core/mcp_server.py，已设计的 MCP 工具：

agent_bridge.list_rooms agent_bridge.get_current_turn agent_bridge.read_messages agent_bridge.get_agent_pending agent_bridge.reply_turn agent_bridge.send_message

注意：当前为自实现 stdio JSON-RPC MCP-like server，后续仍需使用真实 MCP client 做兼容性验证。

3.7 Agent Skill 文档 已完成

新增 core/skills/agent-bridge-room-participant/SKILL.md，约束 Agent 行为：必须通过 MCP reply_turn 或 HTTP callback 写回 Agent Bridge，必须携带 turn_id 和 correlation_id，不能伪造身份或直接修改 active.jsonl。

3.8 UI 产品化改造 已完成

UI 从"开发者调试配置面板"进化为更适合普通用户操作的产品界面。

面向用户

  • Agent 简洁配置卡片
  • 显示名称 / 唯一标识
  • Agent 类型选择
  • 连接状态
  • 测试连接 / 发送测试消息
  • 保存后的下一步提示

折叠至高级

  • 高级设置
  • 开发者配置
  • Adapter 参数
  • Auth 配置

3.9 测试连接与完整联调区分 已完成

测试连接

只验证 OpenClaw Gateway / Webhook 地址是否可达。成功不代表 callback 已完成。

发送测试消息

走 Runtime V2，写入测试消息，唤醒目标 Agent，等待 callback / sync_response。

3.10 自动测试聊天室 已完成

当 Agent 未加入任何 room 时，点击"发送测试消息"可自动创建临时测试聊天室 test_{agent_id}_{random}，标记为 meta.temporary，UI 显示"测试房间"并提供"清理"按钮（先暂停再删除）。

4. 推荐的真实联调流程

安装 / 启动

git checkout dev && git pull
bridge start
# 或开发模式: python cli/bridge start
# 打开 http://127.0.0.1:7899

OpenClaw 联调步骤

  1. 打开 Agent 页面
  2. 点击扫描本机 Agent
  3. 找到 OpenClaw 并添加到 Agent Bridge
  4. 确认配置：显示名称 / 唯一标识 openclaw / 类型 OpenClaw / 会话 agent:main:main
  5. 点击保存
  6. 点击测试连接 — 确认 Gateway 可达
  7. 点击发送测试消息 — 确认自动创建测试聊天室
  8. 观察 OpenClaw 是否收到包含 callback_url 的消息
  9. 观察 UI 是否收到 callback 回写
  10. 观察测试房间消息、events、runtime log
  11. 完成后清理测试房间

6. 当前风险点

6.1 OpenClaw 是否真的能 callback 仍需实测 最大风险

代码已把 callback_url 放进 OpenClaw message，但实际执行取决于：OpenClaw session 工具权限、exec/http 请求是否需用户批准、127.0.0.1:7899 是否可达、运行环境是否同机、callback token 是否正确。

6.2 MCP Server 兼容性风险

当前为自实现协议，风险包括：initialize 协议细节不完整、tools/list schema 不完全兼容、tools/call 返回格式差异、stdio flush/newline/error code 处理。

6.3 UI 与后端配置字段需持续保持一致

adapter schema 变化时，必须同步更新：UI collectAdapterData、server handle_update_config_full、adapters normalize_adapter、tests。

6.4 自动测试房间会写入 bridge.yaml

不清理会积累测试 room，多次测试同一 Agent 会生成多个 test_xxx，用户可能误把测试房间当正式房间。可通过"复用同一 Agent 的测试房间"优化。
